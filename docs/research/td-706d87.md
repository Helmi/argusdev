## 2. Architectural Question Resolution

The task spec raised three questions before any hook work should proceed. Answering them first:

### Q1: How does ArgusDev create a Cline session?

`src/adapters/cline.ts` sets `command: 'cline'`. This is the Cline CLI binary (`npm i -g cline`), **not** a VS Code extension. ArgusDev spawns it as a PTY process via `node-pty`, exactly the same as it does for Codex, Gemini, Claude Code, etc.

**VERIFIED** — `cline.ts` adapter source.

### Q2: Does ArgusDev own the Cline process?

Yes. ArgusDev's `SessionManager` spawns the `cline` CLI binary in a PTY. It controls its lifecycle. The VS Code extension concern in the task spec is a red herring — the CLI is a fully independent process that shares `Cline Core` with the extension but runs standalone.

**VERIFIED** — `src/adapters/base.ts` + `SessionManager` spawn pattern.

### Q3: Can ArgusDev write to `.clinerules/hooks/` and trust Cline to pick it up?

Yes, with a better option: **use the `--hooks-dir` CLI flag** instead. `cli/src/index.ts` exposes `--hooks-dir <path>` which calls `setRuntimeHooksDir()`. This directory is searched first in `getAllHooksDirs()`, before global and workspace hooks. ArgusDev can pass a session-scoped directory (e.g. `<configDir>/cline-hooks/<sessionId>/`) containing the hook scripts, without touching the user's worktree at all. `--hooks-dir` adds to the search path — it does not replace global or workspace hooks, so user-owned hook scripts coexist cleanly.

The flag is defined on both the bare `cline` invocation (line 1184) and the `cline task` subcommand (line 973) — ArgusDev's current `command: 'cline'` spawns the bare invocation, which is covered.

The worktree `.clinerules/hooks/` path still works as a fallback, but `--hooks-dir` is cleaner and avoids worktree pollution.

**VERIFIED** — `cli/src/index.ts` lines 973 and 1184 (both default and task subcommand) + `cli/src/agent/ClineAgent.ts` line 144 + `src/core/storage/disk.ts` `getAllHooksDirs`.

## 3. Cline Hooks Audit

### 3.1 Full Hook Event List

Source: `src/core/hooks/hook-factory.ts` `Hooks` interface (ground truth, supersedes documentation):

| Hook Name | Fires when | ArgusDev state | Notes |
|---|---|---|---|
| `TaskStart` | New task begins | `idle` | Confirms hooks wired; matches Codex `SessionStart` pattern |
| `TaskResume` | Interrupted task resumes | `busy` | User clicked resume |
| `TaskCancel` | User cancels task | `idle` | Non-cancellable itself |
| `TaskComplete` | `attempt_completion` succeeds | `idle` | Primary idle signal |
| `PreToolUse` | Before each tool executes | `busy` | Main busy signal |
| `PostToolUse` | After each tool executes | (informational) | Stays busy until next event |
| `UserPromptSubmit` | User sends a message | `busy` | Redundant with PreToolUse but useful for pure-chat |
| `Notification` | Every `ask` (except `command_output`) | `waiting_input` | **Critical — see §3.3** |
| `PreCompact` | Before context window compaction | (informational) | Not relevant for state |

**9 events total.** Official documentation (as of April 2026) lists 8 — `Notification` is **not documented** publicly but is implemented in source and used internally.

**CONFIDENCE: VERIFIED** for all 9 events (source code `hook-factory.ts`).

### 3.2 Hook Mechanism

- **Hook type:** Shell script (extensionless on macOS/Linux, `.ps1` on Windows)
- **Script name = hook name**, e.g. a file named `PreToolUse` with execute permission
- **Input:** JSON via stdin (common fields: `hookName`, `clineVersion`, `timestamp`, `taskId`, `workspaceRoots`, plus hook-specific data)
- **Output:** Single-line JSON on stdout: `{ "cancel": false, "contextModification": "...", "errorMessage": "" }`
- **Timeout:** 30 seconds per hook execution
- **Transport:** Shell process (like Codex/Gemini approach — `curl` needed, no native `http` type)
- **Platform:** macOS and Linux only (Windows `.ps1` supported but not relevant here)

**CONFIDENCE: VERIFIED** (source: `hook-executor.ts`, `hook-factory.ts`, official docs).

### 3.3 `Notification` Hook — The waiting_input Signal

This is the key finding. From `src/core/task/index.ts`:

```typescript
// Every ask() call except command_output emits:
await NotificationHook.emitUserAttentionNotification(context, {
  source: type,    // ask type: "tool", "followup", "command", etc.
  message: text,
})
```

From `notification-hook.ts`, this builds a `NotificationData` with:
- `event: "user_attention"`
- `waitingForUserInput: true`
- `requiresUserAction: true`

This fires **before** `pWaitFor()` — i.e. before the process blocks waiting for user response. This is ArgusDev's `waiting_input` signal.

The task spec mentioned `onNotification` as a speculative event name. The actual implementation uses `Notification` (the hook name) with `event: "user_attention"` in the payload.

**CONFIDENCE: VERIFIED** (source code `task/index.ts` + `notification-hook.ts`).

### 3.4 Idle State Coverage

`TaskComplete` fires in `AttemptCompletionHandler.ts` before the "Start New Task" UI renders. This is the per-response idle signal (analogous to `agentStop` in Copilot CLI or `Stop` in Claude Code).

`TaskCancel` fires on user cancellation — also transitions to `idle`.

There is **no generic "response done, back to prompt" idle event** separate from `TaskComplete`. If the agent loops without calling `attempt_completion` (e.g. asks a followup question mid-task), the session transitions to `waiting_input` via `Notification`, then back to `busy` on the user's response — no intermediate `idle` between turns.

**CONFIDENCE: VERIFIED** (source code).

### 3.5 ArgusDev State Mapping

| ArgusDev state | Hook | Notes |
|---|---|---|
| `busy` | `UserPromptSubmit` + `PreToolUse` | Both needed: UserPromptSubmit for pure-chat, PreToolUse for tool execution |
| `idle` | `TaskStart` + `TaskComplete` + `TaskCancel` | `TaskStart` → initial idle; `TaskComplete` → per-response idle; `TaskCancel` → abort idle |
| `waiting_input` | `Notification` (event=`user_attention`) | Fires for every `ask` except `command_output` |

### 3.6 Hook Invocation for ArgusDev

Each hook script receives JSON via stdin and must POST to the ArgusDev hook-state endpoint:

```bash
#!/bin/bash
# ~/.config/argusdev/cline-hooks/<sessionId>/PreToolUse
curl -s -X POST "http://127.0.0.1:<port>/api/internal/sessions/<sessionId>/hook-state/busy" \
  > /dev/null 2>&1 || true
echo '{"cancel":false}'
```

The `echo` output is required — Cline validates the JSON response schema (empty stdout is tolerated by `validateHookOutput` as long as it doesn't trigger schema errors, but an explicit response is safer).

Scripts must be **executable** (`chmod +x`).

### 3.7 Hooks Enabled by Default

`getHooksEnabledSafe(userSetting)` returns `true` when `userSetting` is `undefined` (i.e. no config override). Hooks are **on by default** in both CLI and extension. The `--hooks-dir` flag adds a directory; no extra flag needed to enable hook execution.

**CONFIDENCE: VERIFIED** (`hooks-utils.ts`).

## 5. Recommendation

**PROCEED**

All three ArgusDev states (`busy`, `idle`, `waiting_input`) have hook coverage:

- `busy` → `UserPromptSubmit` + `PreToolUse` (VERIFIED)
- `idle` → `TaskComplete` + `TaskCancel` (VERIFIED)
- `waiting_input` → `Notification` with `event: "user_attention"` (VERIFIED in source; untested live)

The hook transport is shell-script + curl, matching the Codex/Gemini pattern in ArgusDev.

**Implementation path:**

1. Add `generateHookConfig()` to `ClineAdapter` — write executable scripts into a session-scoped directory (e.g. `<configDir>/cline-hooks/<sessionId>/`), one script per hook event.
2. Return `{ argsToInject: ['--hooks-dir', hooksDir] }` from `generateHookConfig()` — no worktree pollution.
3. On session cleanup, delete the session-scoped hooks directory.
4. Retain `detectClineState()` as fallback (PTY regex) for environments where hooks fail silently.
5. Mark `hookBasedDetection: true` per state once smoke-tested live.

The existing PTY detection (`detectionStrategy: 'cline'`) should remain active but deprioritized once hook coverage is confirmed — same hybrid approach used for Claude Code.

## 6. Open Questions

1. **`Notification` in yolo/headless mode:** Does `Notification` fire when Cline is invoked with `-y` (auto-approve)? In yolo mode, `ask()` calls may short-circuit before the `pWaitFor` block. If Cline auto-approves, the `waiting_input` hook would still fire (notification is emitted before the `pWaitFor`), but the practical state transition would be `waiting_input → busy` in milliseconds. This is acceptable — same pattern as Claude Code's permission_prompt in auto-approve. Needs live test to confirm.

2. **Output JSON requirement:** The `validateHookOutput` in source rejects deprecated fields but tolerates empty stdout per the schema. The safest approach is to always output `{"cancel":false}` from ArgusDev hook scripts (same as other adapters do implicitly). Confirm this doesn't cause visible "hook failed" UI in Cline's TUI.

3. **`PostToolUse` state:** `PostToolUse` fires after tool completion but before the next `PreToolUse`. The agent is still processing during this window. ArgusDev should either ignore `PostToolUse` (stay `busy`) or skip it entirely. Not hooking it is the correct choice — same as the Codex adapter which skips `PostToolUse`.

4. **Multiple `attempt_completion` in one task:** If an agent calls `attempt_completion`, gets rejected, and continues — does `TaskComplete` fire on every accepted completion or only on the first? `AttemptCompletionHandler` fires it unconditionally on success. ArgusDev correctly transitions to `idle` each time, which is correct behavior.

5. **`TaskCancel` timing:** `shouldRunTaskCancelHook()` has guard logic in `task/index.ts` — there are cases where `TaskCancel` does NOT run (race conditions, already-aborted tasks). Idle detection via `TaskCancel` alone is insufficient. `TaskComplete` is the reliable idle signal; PTY fallback handles edge cases.

6. **PTY regex validity for Cline CLI 2.0:** The existing `detectClineState()` regex in `stateDetection.ts` was written against an older Cline TUI. Cline CLI 2.0 shipped a redesigned interactive mode. The patterns (`[act|plan] mode`, `cline is ready for your message`) may or may not match the current TUI output. Since the PTY regex is retained as fallback, the implementation PR should verify the patterns against a live Cline 2.0 session — outside the scope of this hook research.
