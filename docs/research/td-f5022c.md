## 3. Hooks Audit

### 3.1 Event surface

Documentation at `https://docs.factory.ai/reference/hooks-reference.md` lists **9 hook events**:

| Event | When it fires | ArgusDev state | Notes |
|---|---|---|---|
| `SessionStart` | Session starts or resumes | `idle` | Also fires on `/clear` and `/compact` |
| `UserPromptSubmit` | User submits a prompt, before Droid processes | `busy` | No matchers |
| `PreToolUse` | Before a tool call executes | `busy` | Matchers supported (exact, regex, wildcard) |
| `PostToolUse` | After tool call completes successfully | — | Informational; not needed for state |
| `Notification` | Permission requests **and** 60-second idle prompts | **ambiguous** — see §3.3 | No matchers, no type field in payload |
| `Stop` | Droid finishes responding (not on user interrupt) | `idle` | Per-response, not per-session |
| `SubagentStop` | A sub-droid task finishes | — | Informational |
| `PreCompact` | Before context compaction | — | Not relevant for state |
| `SessionEnd` | Session terminates | `idle` | Terminal signal; `Stop` covers per-turn idle |

**Correction from task spec:** The spec estimated 2 confirmed events. Research confirms 9. The hook surface is significantly richer than assumed.

### 3.2 Config format

Hook type: **`command` only** (no `http` type). Hooks invoke shell commands and receive JSON via stdin.

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -X POST http://127.0.0.1:<port>/api/internal/sessions/<id>/hook-state/busy > /dev/null 2>&1 || true"
          }
        ]
      }
    ]
  }
}
```

Matchers are supported for `PreToolUse` and `PostToolUse` only. Events without matcher support (`Notification`, `UserPromptSubmit`, `Stop`, `SubagentStop`) use bare hook arrays (no `matcher` field).

### 3.3 Config file location

Three locations, layered (later overrides earlier):
- `~/.factory/settings.json` — user-level (all projects)
- `.factory/settings.json` — per-project (version-controlled)
- `.factory/settings.local.json` — per-project local (not version-controlled)

**Critical:** These are full settings files, not dedicated hooks files. They contain models, MCP servers, tool permissions, and hooks under a single `hooks` key. The `.codex/hooks.json` pattern (dedicated hooks file, safe to overwrite) does **not** apply here. The correct template is `writeGeminiHookFiles` — merge ArgusDev hooks into the existing settings object, strip on cleanup. Using a simple overwrite would clobber user MCP config and preferences.

### 3.4 Notification event — the ambiguous case

The `Notification` event fires for **both** permission requests (Droid asking the user to approve a tool action) and idle prompts (Droid notifying after 60+ seconds of inactivity). The docs explicitly state: *"Notification events trigger on permission requests and idle states (60+ seconds)."*

Payload:
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "off",
  "hook_event_name": "Notification",
  "message": "<notification message text>"
}
```

**There is no `type` or `subtype` field.** The only discriminator is the `message` string. Matchers are not supported for Notification.

This creates a problem for ArgusDev's `waiting_input` state:

- A naive `Notification → waiting_input` mapping fires on *both* permission requests and idle notifications. If `Stop → idle` already ran (correct), an idle-prompt Notification 60 seconds later would incorrectly flip the state back to `waiting_input`.
- A message-parsing approach (`jq` + grep to only POST `waiting_input` when message looks like a permission request) is feasible but brittle — the exact message strings for permission requests are not documented, can change across Droid versions, and were not smoke-tested.
- The conservative approach — skip `Notification`, retain PTY fallback for `waiting_input` — avoids false positives at the cost of slower detection. This is the same decision made for Gemini (see `buildGeminiHookConfig` comment: `Notification` scoped to `ToolPermission` via matcher, which Droid lacks).

**Recommendation: skip `Notification → waiting_input` for the initial implementation. Keep PTY fallback active for `waiting_input`. Revisit if Factory adds a subtype field or matcher support for Notification.** This matches the Gemini partial-hook pattern (`hookBasedDetection = true` for busy/idle, PTY fallback retained for `waiting_input`).

## 5. Recommendation

**PROCEED** with hook-based implementation, using the Gemini-style merge pattern.

### State mapping

| Event | ArgusDev state |
|---|---|
| `SessionStart` | `idle` |
| `UserPromptSubmit` | `busy` |
| `PreToolUse` | `busy` |
| `Stop` | `idle` |
| `SessionEnd` | `idle` |
| `Notification` | **not mapped** — PTY fallback retained for `waiting_input` |

### Implementation approach

- Config target: `.factory/settings.json` in the worktree (per-project), **not** `~/.factory/settings.json` (global).
- Pattern: follow `writeGeminiHookFiles` — load existing settings, strip stale ArgusDev hook entries (identified by the `/api/internal/sessions/` URL marker), merge new entries, write back. Backup pristine settings on first session; restore on cleanup.
- Hook command pattern (matches Codex/Gemini):
  ```
  curl -s -X POST http://127.0.0.1:<port>/api/internal/sessions/<id>/hook-state/<state> > /dev/null 2>&1 || true
  ```
- `hookBasedDetection = true` for busy and idle. `waiting_input` uses PTY fallback.
- `PreToolUse` can be scoped with a wildcard matcher (`*`) to cover all tools, consistent with other adapters.
- No `generateHookConfig` currently exists in `DroidAdapter`. Add it following `CodexAdapter.generateHookConfig`.

### Key difference from Codex

Codex writes a dedicated `.codex/hooks.json` (overwrite-safe). Droid's `.factory/settings.json` is a full settings object shared with MCP, models, and permissions config. An overwrite would destroy user settings. The Gemini merge pattern is required.

## 7. Open Questions

1. **PTY hook execution:** Do Droid hooks fire reliably when `droid` runs inside ArgusDev's PTY session (not just in a standard terminal)? Not tested. Recommend smoke-testing before shipping: install Droid, write a minimal `.factory/settings.json` with a `Stop` hook, run a session in PTY, confirm curl fires.

2. **`Notification.message` strings:** What exact message text does Droid send for permission-prompt notifications vs idle notifications? If `factory` adds documentation or if a future test reveals stable patterns, a `jq`-based message filter could be added to route `waiting_input` via hooks rather than PTY fallback.

3. **`SessionEnd` deduplication:** `Stop` fires per-response (idle) and `SessionEnd` fires on termination (also idle). Both mapping to `idle` is correct but redundant. No problem in practice — idempotent transitions are fine — but worth noting.

4. **Per-project `.factory/settings.json` vs global `~/.factory/settings.json`:** The adapter must write to the worktree's `.factory/settings.json`, not the global file. Verify that Droid loads project-level settings correctly when invoked from within the worktree directory. Docs confirm per-project loading, but not smoke-tested.

5. **`droid` binary location at session start:** The existing `DroidAdapter` uses `command: 'droid'`. Confirm this resolves in ArgusDev's PTY environment path at session creation time.
